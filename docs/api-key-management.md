# API Key Management

## What it is

Long-lived API keys for machine-to-machine authentication. Keys are generated by superusers,
stored as SHA-256 hashes, and used via `Authorization: Bearer <key>` headers.

## Why it's needed

Studio Mode and future integrations (CLI tools, external webhooks, CI pipelines) need to
authenticate against the API without user session cookies. Embedding user passwords in
scripts is insecure. Long-lived API keys scoped to a role provide a safe alternative.

## Security design

- The raw key is a randomly generated 32-byte hex string (256 bits of entropy).
- Only the SHA-256 hash is stored. The raw key is returned exactly once at creation and
  never again.
- Authentication: incoming `Bearer <key>` requests are authenticated by hashing the provided
  value and doing a constant-time comparison against the stored hash.
- `last_used_at` is updated on each successful use (useful for auditing stale keys).

## Schema

```sql
CREATE TABLE api_keys (
  id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  label        TEXT        NOT NULL,
  key_hash     TEXT        NOT NULL UNIQUE,
  created_by   UUID        NOT NULL REFERENCES users(id),
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_used_at TIMESTAMPTZ
);
```

## Endpoints

| Method   | Path                  | Auth      | Purpose                                 |
| -------- | --------------------- | --------- | --------------------------------------- |
| `POST`   | `/api/admin/keys`     | superuser | Generate a new key                      |
| `GET`    | `/api/admin/keys`     | superuser | List keys (metadata only, no raw value) |
| `DELETE` | `/api/admin/keys/:id` | superuser | Revoke a key                            |

## Audit logging

All key operations (create, revoke) are written to the audit log with `actor_id` and the
key's `id` (never the raw value).

## Dependencies

- **Superuser bootstrap** (`docs/superuser-bootstrap.md`) — key creation is superuser-only.
- **Audit logging** (`docs/audit-logging.md`) — key operations are audited.
